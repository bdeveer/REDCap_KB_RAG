REDCap BRANCHING LOGIC ANALYSIS
================================

EXPRESSION SUBMITTED:
[age] >= 18 && [consent_status] == 'yes' && [site] != 'withdrawn'

STATUS: OPERATOR ERROR IDENTIFIED
Severity: Critical - Field will not display as expected

ISSUE IDENTIFIED:
================

The problem is with the string comparison operators in this expression. In REDCap branching logic, the correct operators for string comparisons are:

  INCORRECT: == 'yes'    (using single-quoted string with ==)
  INCORRECT: != 'withdrawn'  (using single-quoted string with !=)

  CORRECT: = 'yes'    (using single = for equality)
  CORRECT: <> 'withdrawn'  (using <> for inequality)

WHY THIS MATTERS:
=================

REDCap makes a clear distinction between:
- Numeric comparison operators: ==, !=, >=, <=, >, <
- String comparison operators: =, <>

The double equals (==) and not-equals (!=) operators are designed for numeric/field comparisons, not string literals. When you use == with a quoted string like 'yes', REDCap may not properly evaluate the condition, causing the field to fail to display.

CORRECTED EXPRESSION:
====================

[age] >= 18 && [consent_status] = 'yes' && [site] <> 'withdrawn'

Breaking down the corrections:
- [age] >= 18  ✓ (unchanged - numeric comparison is correct)
- [consent_status] = 'yes'  ✓ (changed from == to =)
- [site] <> 'withdrawn'  ✓ (changed from != to <>)

RESULT:
=======

After making these changes, the field should display when:
1. Age is 18 or older, AND
2. Consent status is exactly 'yes' (as text), AND
3. Site is not 'withdrawn'

ADDITIONAL NOTES:
=================

- Make sure that 'yes' and 'withdrawn' match the exact values stored in your fields (including case)
- If consent_status is a dropdown with internal coding (e.g., coded as 1='yes'), you may need to compare against the code (1) instead of the label ('yes')
- Test the expression in the Branching Logic Tester in REDCap to verify it works as expected
- The logical operators (&&) are correct and don't need to be changed
